作成日: 2026-06-03

このドキュメントは `docs/04_Codex_after_UI_function_implementation_prompt.md` 実装後に残っている機能を、実運用に近づけるための優先順位で整理したものです。

## 前提

現在のSlideRoomは、ルーム作成、参加、ファイル提出、スライド順番調整、設定保存、PPTX出力まで動くMVPです。

PPTX出力は、アップロード済みPPTXだけで構成された結合対象ではブラウザ内OOXMLコピー方式の実PPTX結合を試行します。対象外ファイルを含む場合や結合失敗時は、既存の `pptxgenjs` 簡易PPTXへfallbackします。

実サービスとして次に最優先で解消すべき差分は、fixture/手動検証による品質担保、複雑なPPTXでの破綻パターン整理、fallback時の詳細表示です。

### P1-2. PDF出力を本物にする

現状:
基本実装済み。アップロード済みPPTXだけで構成された結合対象では、ブラウザ内でPPTXをOOXML解析してSVGプレビュー化し、そのスライド画像を `jspdf` でPDFへ配置する。これにより、従来のプレースホルダーBlobではなく実PPTX由来のPDFを出力できる。対象外ファイルや保存BlobのないPPTXを含む場合は、嘘のPDFを生成せずエラーにする。

制限:
- PDFはスライドを画像として配置するため、PDF内テキスト検索・選択はできない
- 複雑なグラフ、アニメーション、特殊効果、未対応OOXMLはプレビュー同様に崩れる可能性がある
- 本番品質の高忠実度変換は、サーバーサイドLibreOffice変換が引き続き優先候補

タスク:
- PPTX結合後にPDFへ変換する方式を決める（短期はブラウザ内SVGラスタライズ方式で実装済み）
- サーバーサイドLibreOffice変換を優先候補にする（本番高忠実度向けに継続検討）
- PDF出力履歴とエラー表示を整える（成功・失敗履歴は実装済み）

完了条件:
- 実PPTX由来のPDFが出力できる（基本実装済み。実PPTX 2件6枚で検証済み）

## 次に着手する推奨タスク

次は **P0-4. PPTX/PDF出力の検証データとテストを作る** から始める。

理由:
- PPTX結合、PPTXプレビュー、PDF出力の基本実装が入ったため、ここからは固定fixtureで壊れない保証を作る必要がある
- PDFはプレビューSVG由来なので、プレビュー改善がPDF品質にも直結する
- テストがないまま複雑なPPTX対応やUI改善へ進むと、以前のように「開けないファイル」やレイアウト崩れを再発しやすい

最初の実装単位:

1. `fixtures/pptx/simple-a.pptx` と `simple-b.pptx` を作る
2. 実PPTX結合結果をZIPとして検査するテストを追加する
3. 実PDF出力結果のPDFヘッダー、ページ数、画像数、プレースホルダー文字列なしを検査する
4. 生成PPTXに「MVP簡易スライド」などのfallback文言がないことを確認する
5. PowerPointまたはLibreOfficeで手動オープン確認する
